## Problem Statement

同时使用多个 coding CLI agent（Codex CLI / Claude Code / Gemini CLI）已是主流工作方式（"agentmaxxing"），但协调它们的方式原始：多开终端窗口手动盯、手动复制粘贴上下文交接、agent 的口头汇报无法核实。2026-07 已调研的同类工具（Claude Squad、Agent Deck、agent-session-manager、Conduit、ntm）均基于 tmux —— 在 Windows 上必须 WSL —— 且协同深度停留在"并行 + worktree 隔离"，没有结构化交接，也没有对 agent 声明的验证。

AgentRoom 要做的是这些工具上面缺失的一层：一个本地 TUI"教室"，让用户像老师一样看到每个座位（CLI 实例）在干什么、谁举手要审批，用 `@座位` 派活，用带证据的 context pack 让座位互相评审 —— 并且（阶段 3a）对 agent 的关键声明自动重放验证。

## What Makes This Cool

- **证据当裁判，不是档案柜**：agent 说"测试过了"，AgentRoom 在 worktree 里重跑测试命令自己打 `verified/rejected` 戳。黑板从笔记本变成测谎仪 —— 已调研的工具中没有一家做这件事。
- **座位成绩单**：verified/rejected 累积成 per-seat 战绩（"本仓库里 Codex #1 十次修复八次可验证"），派活推荐基于本仓库实测战绩而非印象。
- **tournament 模式**：同一任务派给两个座位并行 worktree，第三个座位评分 —— `Assignment.inferredIntent` 里的 `compare` 早已定义，只差开发。
- **`agentroom replay <session>`**：events.jsonl + evidence 天然构成整个教室的语义级 asciinema，是最好的开源传播物料。
- **原生 Windows**：无头子进程走管道，绕开 tmux/ConPTY，已调研竞品在 Windows 上都要 WSL，AgentRoom 不用。

## Constraints

- Solo 开发者，学习节奏（60 天 Agent 全栈转型计划并行），可投入时间碎片化。
- Windows 原生一等公民；禁止依赖 tmux / PTY。
- AgentRoom 自身不接任何 LLM API（零 API 成本、零 key 管理）。
- 技术栈按原架构文档：TypeScript + Node.js + Ink，文件/JSONL 存储，无数据库。
- 用户学习规则（AI 辅助 30% / 手写核心 70%）：脚手架、类型定义、配置可由 AI 生成；ACP 会话管理、corridor 编排、context pack 组装等核心逻辑由用户手写。

## Premises

1. **（2026-07-03 与用户逐条脑暴后重写）V1 = 多工具同屏的单 CLI 教室。** Codex / Claude / Gemini 作为座位在同一个 TUI 里同时可见、可 `@派活`、可互相交接 —— 这是第一版的验收线，不是后续增强（即 product-spec 的 Main Screen 原样是 V1 目标画面）。深度协同（证据黑板 / claim 验证）是 V1 之后的护城河。竞争定位（多开管理红海、深度协同空白）降级为参考信息，不再作为砍 V1 范围的依据。
2. **（2026-07-03 与用户对齐后重写）V1 按 architecture.md M7 原案：runner = 黑盒无头子进程**（`claude -p`、`codex exec`、gemini 非交互模式等，走管道读输出，Windows 原生无 PTY/tmux）。代价按原文档接受："Deep structured integrations can come later"，V1 座位状态较粗（running/done/failed + 文本活动行）。ACP（Agent Client Protocol，三家均已兼容的结构化会话协议）仅记档为"以后深度集成"的候选，不进 V1、不影响 V1 设计。
3. **风险重排：走廊第一（用户已确认，2026-07-03）**。先打通"1 个座位：派发 → worktree 执行 → diff + summary 回收"的最小闭环（证明三家 CLI 能被程序化驱动、Windows 原生可用），再搭任何脚手架/registry/TUI 皮肤。用户表态"先跑通"—— 验证生死题优先于视觉体验。
4. **（经交叉模型质疑后修订）** summary 由 runner agent 依"摘要契约"（schema 见下文《摘要契约 v0》）顺带产出，同时必须配**确定性兜底摘要器**，防止 runner 不守契约时交接静默降级为裸 diff。验收判据（per-CLI，不做聚合）：**每个 CLI ≥10 次派发（三家共 ≥30 次），schema 合规率 ≥85%/CLI**；同一实验同时测量 **claims 产出率**（含可重放 command 的 claim 占比），作为进入阶段 3a 的 go/no-go，判据预先承诺：**至少一个 CLI 在 ≥50% 的派发中产出 ≥1 条可重放 claim 即 go**（裁判系统只需一个可靠供给方即可 demo）；不达则先改契约提示词重测一轮，仍不达则推迟 3a —— 因为兜底摘要器能救交接摘要，但按构造无法生成可重放 claims，claims 供给不足会让裁判系统饿死。某 CLI 不达标：该 CLI 交接走兜底-only，裁判功能对其降级，不影响其它 CLI。
5. **Windows 原生一等公民** —— 同时是与全体 tmux 系竞品的硬差异。

**MVP 明确砍掉/推迟**（推迟到裁判系统验证之后）：hooks 的 4 种运行时、**registry 全量（含 builtin，不只是 3 家工具导入）**、**agent/skill 自动匹配推断**（原 M6/P7 的一部分）、flow 系统、memory 审批队列、配色体系完整版。

## 摘要契约 v0（最小 schema 草案）

runner 完成任务时在 **worktree 根**写 `AGENTROOM_SUMMARY.md`（头部 YAML front-matter，如下），corridor 收尾时将其收编为座位目录的 `summary.md` —— runner 从不直接写座位目录，与阶段 1 权限策略（worktree 外写入一律拒绝）自洽：

```yaml
summary: 一句话说明改了什么
changed_files:
  - src/auth/session.ts
tests:                      # runner 实际执行过的测试/构建命令；没跑则省略
  - command: npm test
    exit_code: 0
claims:                     # 可选但裁判系统依赖它；每条必须可重放
  - text: 修复了会话超时未清理的问题
    command: npm test -- session.test.ts
    expected_signal: exit 0
```

**确定性兜底摘要器**数据源（corridor 阶段自身不主动执行测试）：`git diff --stat`、变更文件清单、进程退出码；若某 runner 的无头模式提供结构化输出（如 `claude -p --output-format stream-json` 的工具事件行），则一并提取其中的测试/构建命令及退出码。兜底产物只含 `summary/changed_files/tests`，**不含 claims** —— 这正是 claims 产出率必须单独测量的原因。契约 schema 的定稿是 Next Steps 第 3 步的正式交付物之一。

## Cross-Model Perspective

Codex 超时（5 分钟无响应），按预案降级为全新上下文的 Claude subagent 冷读（未见脑暴对话，只读磁盘上的四份设计文档 + 结构化背景）。其要点：

1. **最酷版本**：把 evidence 从"filing cabinet"升级为"referee" —— `Claim.status` 有 `verified/rejected` 枚举但四份文档没有任何状态变化机制；应让每个 claim 携带可重放检查（command + 预期信号），AgentRoom 自动重跑打戳。由此自然长出成绩单、tournament（`compare` intent 已定义未开发）、`agentroom replay`。
2. **最暴露兴奋点的一句**："Color is part of the product experience."（product-spec 第 97 行）—— 四套颜色分类法在写下任何代码之前被完整定义，而声称的差异化（claim 验证）零机制。里程碑顺序（TUI 在 M5、evidence 在 M8、验证从未出现）印证了优先级倒挂。
3. **50% 现成件**：`@zed-industries/agent-client-protocol`（npm，Zed 官方 TS 库）—— 类型化 JSON-RPC：spawn、session/new、session/prompt、流式 session/update、session/request_permission。等于原 Milestone 7 + 适配器层大部 + waiting_user + 活动流，三个 CLI 通用且 Windows 原生。剩下的 50% 恰是文档声称的差异化：dispatch 解析、context pack、evidence 存储与验证、blackboard、worktree 策略、Ink TUI。次级矿：Vibe Kanban 遗骸（MIT，Rust）的 executor 归一化与 worktree 管理值得读源码（不复用代码，学 per-CLI 怪癖）。
4. **周末构建顺序**：probe（半天，验证 ACP 在原生 Windows 可用）→ corridor（半天）→ contextpack 双座交接（半天）→ 最小 TUI（半天，可选）。
5. **前提质疑**：前提 4 最弱 —— runner 可能跳过 summary.md、schema 漂移、断言不可验证的 claim，旗舰功能静默退化。修复不需要接 LLM：确定性兜底摘要器（本文档已采纳，并按第二轮评审追加 claims 产出率测量）。前提 1/2/3/5 成立，2 与 5 互相强化。
6. **文档漂移警告**：脑暴砍掉的功能在磁盘文档中仍完整成立 —— 写代码前先改文档（详见 Next Steps 第 1 步的完整漂移点清单）。

> 注（2026-07-03 用户裁决）：上述第 3、4 点中的 ACP-first 建议**未被采纳** —— V1 维持 architecture.md 的黑盒无头原案，ACP 仅记档（见前提 ②）。本节保留原文仅作评审过程记录。

## Approaches Considered

> 工作量双口径说明：CC 口径假设 AI 直接写核心代码，与本项目 30/70 学习规则冲突，**仅作能力参照；实际排期以 human 口径为准**。

### Approach A: 走廊优先（最小可行）
以黑盒无头子进程打通 1→2 个座位的完整走廊（派发 → worktree 执行 → diff + summary 回收 → 交接评审），最后补最小 TUI。无 registry / hooks / flows / memory。（评审版曾建议 ACP-first，经用户裁决改回黑盒原案）
- Effort: S（human ~2–3 周 / CC ~1–2 天，见双口径说明）｜Risk: 低-中｜Completeness: 7/10
- Pros: 生死题（能否程序化驱动 CLI）第一天见分晓；每步产出可用之物；Windows 风险最早暴露。
- Cons: TUI 爽感来得晚；砍掉的功能要靠纪律不做。
- Reuses: 原 architecture.md 类型定义按《类型处置表》分级复用；product-spec 的 ASCII 布局。

### Approach B: 蓝图直建（理想架构）
按原 12 milestone / 17 phase 顺序推进，monorepo，registry + hooks + flows + memory 全量。
- Effort: XL（human ~2–3 月 / CC ~2–3 周）｜Risk: 高｜Completeness: 9/10（全，但风险顺序错）
- Pros: 照文档施工省决策；平台完整；适合练工程化。
- Cons: 前六个 milestone 没有能用的东西；生死风险压在 M7；红海部分先建、护城河部分后建；solo 项目动力枯竭的经典死法。

### Approach C: 裁判优先（创意路线）
把"可重放验证"做成产品核心：claim 携带 `command + 预期信号`，AgentRoom 重跑打 verified/rejected 戳；成绩单；tournament；TUI 即记分板。
- Effort: M（human ~1–2 月 / CC ~1 周）｜Risk: 中｜Completeness: 8/10
- Pros: 唯一护城河 + 最强 demo（"agent 谎报被当场抓住"）+ replay 传播物料。
- Cons: 走廊未通之前是空中楼阁，必须以 A 为前置。

## Recommended Approach

**A→C（用户已批准，2026-07-03）**：先 A 验证生死题，跑通后长成 C；B 的平台件推迟到 C 验证护城河之后再评估。

**V1 完成定义（2026-07-03 前提 ① 重写后更新）**：阶段 1 + 阶段 2 合并构成 V1 —— 走廊只是施工顺序上的第一步，V1 交付物是**多工具同屏的教室 TUI**（product-spec Main Screen 跑起来）。阶段 3（裁判）是 V1 之后的护城河，Approach A 原文里"TUI 爽感来得晚"的取舍表述作废。

### 阶段划分

- **阶段 1 · 走廊**：ACP probe → 单座 corridor（摘要契约 + 确定性兜底）→ 双座 contextpack 交接 demo。
  - **权限策略（无头模式）**：用各 CLI 自带的非交互权限旗标（如 Claude 的 `-p` 配合权限模式旗标、`codex exec` 的沙箱/审批参数，具体旗标 probe 时确认），把可写范围限定在该次派发的 worktree 内；worktree 隔离是主要安全边界。细粒度的权限交互（`waiting_user` 举手）属于以后的深度集成，V1 不做。
  - **失败路径**：单次派发默认超时 10 分钟（可配置）→ kill 进程树、座位状态 `failed`；创建 worktree 前做脏检测；失败保留现场，成功后手动清理（清理命令推迟）。
  - **测试场与依赖**：阶段 1 选一个小型 Node 目标仓做试验场，worktree 内用 pnpm（硬链接 store，安装成本可控）；跨语言/大仓的依赖策略推迟。
  - **会话布局**：从第一天起使用 architecture.md 的 `.agentroom/sessions/sess_*/{events.jsonl, seats/<seat>/{state.json, patch.diff, summary.md}}` 子集，不另设 `runs/` 目录，避免日后迁移。
- **阶段 2 · 教室**：多座位、`@seat` 派发解析（只解析目标/来源/指令原文，**不做 agent/skill 推断** —— 已列入砍面）、Ink TUI（黑板头 + 座位排 + desk 面板）、stop seat、SeatState 按 probe 实测可观测性收敛枚举（`waiting_user` 依赖深度集成，V1 暂不做）。
- **阶段 3a · 裁判（护城河验证）**：replayable check 机制 + claim 重跑打戳 + 最小 blackboard 视图。**重放安全**：只自动执行白名单命令族（test/build/lint），必须在对应 worktree 内；白名单外命令需一键确认。达成 Success Criteria 中"裁判"两条即算验证通过。
- **阶段 3b · 记分与传播（3a demo 通过后解锁）**：per-seat 成绩单、`compare`/tournament、`agentroom replay`。
- **推迟**：hooks 四运行时、registry 全量（含 builtin）、agent/skill 推断、flow、memory 审批队列、配色体系完整版。

2. 摘要契约 per-CLI 合规率与 claims 产出率实测值 —— 决定各 CLI 的主路径/兜底权重与裁判功能覆盖面。

4. 各 CLI 的非交互权限旗标能否把写入范围可靠限定在 worktree 内（安全边界确认）—— probe 时验证。
5. 12 态 SeatState 中哪些在无头输出里真实可观测（纯文本 vs stream-json 差异）—— probe 后收敛枚举；其余状态归入"以后深度集成"。深度集成阶段是否采用 ACP，届时再评估。

## Success Criteria

- **走廊**：`tsx src/corridor.ts "fix X"` 在原生 Windows（无 WSL）产出 `.agentroom/sessions/<id>/seats/<seat>/{patch.diff, summary.md}` 与会话级 `events.jsonl`；超时失败路径可演示；权限旗标实测能把写入限制在 worktree 内。
- **交接**：第二座位的 review 输出实际引用了第一座位 diff 的具体内容（非泛泛而谈）。
- **教室**：TUI 同屏 ≥2 座位（含至少两家不同工具）实时状态、可 `@派发`、可 stop。
- **裁判（3a gate）**：至少一个 claim 被 AgentRoom 自动重跑并打上 verified/rejected；能演示"agent 谎报测试通过被抓"。
- **合规率**：每 CLI ≥10 次、共 ≥30 次派发，得到 per-CLI 的 schema 合规率与 claims 产出率两组数字。

## What I noticed about how you think

- "Agent text is not fact. Important facts should be backed by evidence." —— 别人都在做"并行"的时候你在想"信任"。这是全场最有味道的产品判断，也是已调研市场中无人做的那一块。
- "Flow is not the product center. It is a repeatable dispatch shortcut." —— 你天然抵抗工作流引擎的引力。这个品类里几乎人人最后都造了个 workflow engine，你在设计阶段就把它按住了。
- 四套颜色分类法在第一行代码之前被完整定义 —— 你在乎"看着一教室 agent 干活"的体感。这是产品心，值得保留；但外部冷读一眼看穿"验证机制零笔墨"，兴奋点会拉着优先级跑 —— 走廊优先是你给自己的纪律。
